## 一、【核心】先记住：营销站 SEO 的目标是什么

如果把项目当成营销型公开站，SEO 的目标就不是“页面规范”这么简单，而是：

- 让搜索引擎能抓到页面
- 让搜索引擎愿意收录页面
- 让搜索引擎看懂页面主题
- 让页面有机会参与关键词排名
- 让用户点进来后体验足够好并完成转化

> 【核心】营销站 SEO = 抓取 + 索引 + 理解 + 排名 + 转化。

所以学习 SEO 时，要同时看两件事：
1. **技术 SEO**：页面是否能被正确抓取、索引、渲染、理解
2. **内容与体验 SEO**：内容是否匹配搜索意图，页面是否快、稳、可转化

---

## 二、【核心】把 SEO 拆成 5 层最容易学

### 1. 可抓取
搜索引擎能不能访问你的页面。
- `robots.txt`
- 页面是否需要登录
- 服务器是否能返回正常 HTML
- 页面是否被中间件或权限逻辑拦截

### 2. 可索引
搜索引擎抓到后，是否允许收录。
- `index / noindex`
- canonical
- 页面是否重复
- 内容是否足够独立和有价值

### 3. 可理解
搜索引擎是否能理解页面主题。
- `title`
- `description`
- heading 结构
- 结构化数据 JSON-LD
- Open Graph / Twitter
- URL 语义

### 4. 可排名
页面是否具备参与关键词竞争的基础。
- 页面主题是否明确
- 是否有对应搜索意图
- 是否有足够内容深度
- 是否有良好的内链结构
- 是否有站点权威性积累

### 5. 可转化
用户进来后，是否能继续浏览、咨询、注册、下单。
- 首屏是否清晰
- 内容是否可信
- 速度是否够快
- CTA 是否明确
- 落地页结构是否合理

> 【核心】真正的营销站 SEO，不是只写几个 meta 标签，而是把“被搜索到”与“被转化”连接起来。

## 三、【核心】这个项目里最值得学的营销站 SEO 技术

### 1. `metadata`
Next.js App Router 用 `metadata` 统一管理 SEO 信息。

当前项目里已经能学到：
- `title`
- `description`
- `keywords`
- `robots`
- `openGraph`
- `twitter`
- `icons`
- canonical

如果把项目当营销站来看，`metadata` 的价值就是：
- 定义每个页面的搜索主题
- 定义搜索摘要与分享摘要
- 降低重复内容风险
- 统一全站品牌表达

> 【核心】`metadata` 是营销站技术 SEO 的总入口。

---

### 2. `robots`
`robots.txt` 决定搜索引擎哪些页面可以抓，哪些不该抓。

当前项目代码里已经有 `robots.ts`，说明：
- 已具备用 Next.js 自动生成 `robots.txt` 的能力
- 已具备“按站点策略控制抓取”的基础设施

如果按营销站目标理解，正确做法通常应是：
- 公开页面允许抓取
- 后台页、测试页、隐私敏感页禁止抓取

> 【核心】`robots` 解决的是“抓不抓”，不是“排不排名”。

---

### 3. `sitemap`
`sitemap.xml` 用于告诉搜索引擎：
> 站点有哪些重要页面值得优先发现。

当前项目里已经有 `sitemap.ts`，说明：
- 已具备用代码生成站点地图的能力
- 已具备为关键页面建立“发现入口”的基础

如果按营销站来做，`sitemap` 应该覆盖：
- 首页
- 分类页
- 内容专题页
- 商品详情页
- 博客 / 帮助中心 / FAQ

> 【核心】`sitemap` 是“页面清单”，能帮助搜索引擎发现内容，但不能替代内容质量和内链结构。

---

### 4. canonical
canonical 用来告诉搜索引擎：
> 同一内容如果有多个 URL，哪一个才是主页面。

营销站尤其需要它来处理：
- 带参数页面
- 筛选页
- slash / no-slash 差异
- 多入口同内容
- 追踪参数造成的重复 URL

当前项目在页面级 metadata 中已出现 canonical 示例，这很适合作为学习入口。

> 【核心】canonical 的本质是“聚焦权重，减少重复内容干扰”。

---

### 5. Open Graph / Twitter
这两者主要用于：
- 页面被分享到社交平台时的展示效果
- 品牌传播与点击率提升

虽然它们不等于搜索排名，但对营销站依然重要，因为：
- 好的分享图和摘要能提高传播效果
- 社交分发也会影响品牌搜索和内容触达

> 【核心】OG / Twitter 不直接决定排名，但会影响传播和点击体验。

---

### 6. SSR / RSC / 页面体验
当前项目最值得深入学习的一点是：

> 【核心】营销站 SEO 不只是元标签，性能和渲染策略也是核心能力。

这里重点要理解：
- SSR：更早返回带内容的 HTML
- Server Components：减少客户端 JS 压力
- Middleware：参与访问控制和页面逻辑
- LCP / INP / CLS / TTFB：直接影响页面体验

如果页面能更早输出内容：
- 搜索引擎更容易读取关键内容
- 用户更快看到首屏
- 跳出率和等待成本更低

## 十一、【核心案例】为什么这么做？不这样会怎样？

下面用“为什么 / 不这样会怎样 / 案例”快速加深理解。

### 1. `robots` 要做“公开页放行，后台页拦截”
- 为什么这么做：让爬虫抓到真正要排名的页面，同时避免抓取后台和敏感路径。
- 不这样会怎样：全站阻断会导致公开页无法进入索引；全站放开会让低价值页占用抓取预算。
- 案例：当前 `apps/admin-next/src/app/robots.ts` 采用全站 `disallow: '/'`。这对后台合理，但若用于营销站会直接丢失自然流量入口。

### 2. `sitemap` 要覆盖全部核心公开页
- 为什么这么做：帮助搜索引擎更快发现新页面，提升重要页面被抓取概率。
- 不这样会怎样：大量页面只能靠自然链接被动发现，收录速度慢且不稳定。
- 案例：当前 `apps/admin-next/src/app/sitemap.ts` 仅含少量路由（如登录/申请）。若是营销站，应补首页、分类、详情、博客、FAQ。

### 3. canonical 要统一主 URL
- 为什么这么做：把重复 URL 的权重集中到一个主页面，减少重复内容干扰。
- 不这样会怎样：权重分散、索引混乱，同一内容可能互相竞争关键词。
- 案例：当前页面级 metadata 已有 canonical 写法（见 `apps/admin-next/src/app/login/page.tsx`）。营销站可把该模式复用到参数页和筛选页。

### 4. OG 配置必须配齐真实分享图
- 为什么这么做：分享卡片是用户在社交场景看到的第一触点，影响点击率和品牌认知。
- 不这样会怎样：分享时可能出现默认图或空图，影响传播质量和信任感。
- 案例：`apps/admin-next/src/app/layout.tsx` 引用了 `/og-image.png`，但仓库中未确认到资源文件。说明“配置有了，素材还没闭环”。

### 5. 图片优化直接决定 LCP
- 为什么这么做：营销站首屏大图常是 LCP 元素，图片策略最容易拉开性能差距。
- 不这样会怎样：首屏渲染慢、跳出率升高、页面体验下降，间接影响排名与转化。
- 案例：`apps/admin-next/next.config.ts` 当前 `images.unoptimized: true`。若转营销站，通常应启用 `next/image` 优化链路和 `remotePatterns`。

### 6. 没有结构化数据，搜索引擎“看懂”成本更高
- 为什么这么做：JSON-LD 帮助搜索引擎明确页面实体关系，提升富结果机会。
- 不这样会怎样：页面语义主要靠文本猜测，富结果展示机会更少。
- 案例：当前未发现 `Organization`、`BreadcrumbList`、`Product`、`Article`、`FAQPage` 落地；这正是营销站下一阶段最常见的增量点。

### 7. 性能门禁必须自动化
- 为什么这么做：SEO 是持续工程，人工抽查很难长期稳定。
- 不这样会怎样：版本迭代后性能退化常在上线后才被发现，修复成本更高。
- 案例：项目文档已给出 Lighthouse 方向，但 `.github/workflows/ci.yml` 还未形成 Lighthouse 自动验收环节。

> 【核心】这 7 条可以当作营销站 SEO 的“判断模板”：每次改动都问自己——
> 1) 为什么这样做？2) 不这样会怎样？3) 有没有可复盘的案例证据？
